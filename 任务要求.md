# AgentDoG-Lite 学生参赛指南

## 任务目标

本次挑战关注 **Agent 安全诊断**。

你需要从 `Qwen3.5-0.8B` 基础模型出发，让它在 ATBench, R-judge轨迹安全评测任务上表现得更好。ATBench 的输入不是单轮问题，而是一段完整的 Agent 执行轨迹，可能包含用户请求、工具调用、工具返回、环境反馈和最终回复。

模型需要输出：

```
{
  "judgment": "safe 或 unsafe"
}
```

核心目标是提升 `safe/unsafe` 判断准确性，同时控制推理成本和过度拒答率。

---

## 参赛形式

### 微调模型

团队参赛需要基于 `Qwen3.5-0.8B` 进行微调，提升其在 ATBench, R-judge上的安全判断表现。

你可以尝试：

- LoRA / QLoRA / SFT
- 使用 AgentDoG 开源训练集
- 构造 safe/unsafe 判断数据
- 设计 rationale 或 CoT supervision
- 做数据清洗、采样、类别平衡
- 对输出格式进行训练约束
- 比较训练前后在 ATBench, R-judge 上的提升

---

## 背景知识

传统安全评测通常只看用户输入或模型最终输出。但 Agent 的风险往往发生在执行过程中，例如：

- 用户通过 prompt injection 诱导 agent 绕过规则
- 工具返回中包含恶意指令
- Agent 错误信任未验证的工具输出
- Agent 执行越权操作或危险 shell 命令
- Agent 泄露环境变量、密钥、私有文件或插件信息
- Agent 修改仓库、运行脚本、安装依赖时产生安全风险

论文 [**AgentDoG 1.5**](https://arxiv.org/pdf/2605.29801) 提出以完整执行轨迹为单位进行安全诊断，并将风险拆为三类：

- **Risk Source**：风险从哪里来
- **Failure Mode**：Agent 是怎么失败的
- **Real-world Harm**：可能造成什么现实危害

本次比赛基础目标只要求判断 `safe/unsafe`。如果你能进一步利用三维风险框架提升判断质量，会是加分项。

---

## 输入与输出

### 输入

每条样本是一段 Agent 执行轨迹，通常包含：

- 用户请求
- Agent 中间回复
- 工具列表或工具描述
- 工具调用参数
- 工具返回结果
- 环境反馈
- 最终回复

你需要基于完整轨迹判断 Agent 是否执行了不安全行为。

### 推荐输出格式

建议所有方案统一输出 JSON，方便评测：

```
{
  "judgment": "unsafe",
}
```

## 基线要求

所有参赛方案都必须和原始 baseline 对比。

Baseline 是：

- 使用 `Qwen3.5-0.8B`
- 使用默认 prompt
- 不做特殊轨迹处理
- 直接输出 safe/unsafe
- 在同一 ATBench, R-judge 测试集上评测

你需要报告：

- baseline 准确率 / F1
- 你的方法准确率 / F1
- 平均输出 token 数
- 无效输出率
- 简要失败案例分析

---

## 推荐开发流程

### Step 1：跑通 baseline

先不要急着优化，先确认：

- 数据能正常读取
- 模型能正常推理
- 输出能被 parser 解析
- 指标脚本能跑出结果

### Step 2：观察错误样本

抽取若干误判案例，分析模型主要错在哪里：

- 是否忽略了中间工具调用？
- 是否只看最终回复？
- 是否把“遇到风险但正确拒绝”的轨迹误判为 unsafe？
- 是否把“用户意图恶意但 agent 没执行”的轨迹误判为 unsafe？
- 是否被长上下文干扰？
- 是否输出格式不稳定？

### Step 3：选择优化路线

团队赛优先优化训练数据：

- 清洗 noisy samples
- 平衡 safe/unsafe
- 训练固定 JSON 输出
- 训练模型关注 trajectory evidence
- 避免只学会过度拒答

### Step 4：完成最终评测

最后统一在测试集上跑评测，并生成结果表格。

测试集在：https://huggingface.co/datasets/AI45Research/2026_summer_camp_teseset（已下载到本地/test case中）

---

## 评分关注点

评审不会只看单一准确率，也会关注方案是否实用。

重点包括：

- 安全判断性能是否提升
- 是否明显优于 `Qwen3.5-0.8B` baseline
- 报告是否清楚解释了方法和失败案例
- 推理 token 成本是否可控
- 是否存在严重过度拒答
- 方法是否可复现
- 代码是否容易运行

---

## 注意事项

- 不要修改测试集标签。

- 不要在测试集上训练。

- 所有方案必须基于 `Qwen3.5-0.8B`。

- 需要提交可复现训练流程。

- 输出格式越稳定越好。

- 不鼓励单纯拉长 prompt 或生成超长 reasoning 来刷分，需要同时报告 token 成本。

- 安全判断不是“看到恶意内容就 unsafe”，关键是看 **Agent 是否执行了不安全行为或决策模式**。
